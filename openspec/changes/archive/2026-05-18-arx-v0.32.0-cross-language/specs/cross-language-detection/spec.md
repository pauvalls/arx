# Cross-Language Detection Specification

## Purpose

Trace dependencies between IDL/schema source files (e.g., `.proto`) and their generated artifacts (e.g., `.pb.go`), giving polyglot repositories visibility into generated-code origins. Existing per-language detectors run first; this domain adds a post-processing phase that links sources to generated outputs via configurable mappings.

## Requirements

### Requirement: Language Field on Dependency

The `Dependency` struct MUST include a `Language` string field. The field MAY be empty for backward compatibility. Each per-language detector SHOULD populate `Language` with its identifier (e.g., `"go"`, `"typescript"`, `"python"`).

#### Scenario: Backward compatible empty field

- GIVEN a project with no `cross_language` configuration
- WHEN `arx check` runs
- THEN all existing `Dependency` records have an empty `Language` field
- AND all existing rules and output formats behave identically

#### Scenario: Detector populates language

- GIVEN a Go detector updated to set `Language`
- WHEN it emits a `Dependency` for `service.go â†’ internal/pb/order.pb.go`
- THEN the dependency carries `Language: "go"`

### Requirement: Cross-Language Config Section

The config MUST support a `cross_language` section with a `mappings` list. Each mapping MUST contain `source_pattern` (glob), `generated_pattern` (glob), and `language` (generated-code language). Multiple mappings SHALL be supported.

#### Scenario: Single mapping

- GIVEN `cross_language.mappings` with `{source_pattern: "proto/**/*.proto", generated_pattern: "internal/pb/**/*.pb.go", language: "go"}`
- AND `proto/order.proto` and `internal/pb/order.pb.go` exist
- WHEN the detector processes this mapping
- THEN the file pair is matched

### Requirement: CrossLanguageDetector Post-Processing

The detector SHALL run after all per-language detectors. For each mapping:

1. Find source files matching `source_pattern`
2. For each source, find generated files matching `generated_pattern` with the same base name
3. Verify the generated file contains a protoc-generated header comment (e.g., `// Code generated by protoc-gen-go`) within the first 5 lines
4. Emit synthetic `Dependency` with `SourceFile=proto path`, `ImportPath=generated path`, `Language="cross"`

Failures for individual matches MUST be silently skipped. Errors MUST NOT block the full check.

#### Scenario: Happy path â€” proto to pb.go detected

- GIVEN a mapping `proto/**/*.proto â†’ internal/pb/**/*.pb.go`
- AND `proto/order.proto` exists
- AND `internal/pb/order.pb.go` exists with header `// Code generated by protoc-gen-go`
- WHEN cross-language detection runs
- THEN a synthetic `Dependency{SourceFile:"proto/order.proto", ImportPath:"internal/pb/order.pb.go", Language:"cross"}` is emitted

#### Scenario: Missing generated file

- GIVEN a mapping matching `proto/order.proto`
- BUT no matching file under `internal/pb/order.pb.go` exists
- WHEN cross-language detection runs
- THEN no dependency is emitted for this pair
- AND the check continues without error

#### Scenario: Non-generated file skipped

- GIVEN `proto/order.proto` and `internal/pb/order.pb.go` both exist
- BUT `order.pb.go` lacks a protoc-generated header
- WHEN cross-language detection runs
- THEN no dependency is emitted for this pair
- AND the check continues without error

#### Scenario: Multiple mappings produce multiple deps

- GIVEN mappings for both `proto/**/*.proto â†’ internal/pb/**/*.pb.go` and `proto/**/*.proto â†’ frontend/src/**/*.pb.ts`
- AND matching files with valid headers exist for both
- WHEN cross-language detection runs
- THEN two synthetic dependencies are emitted

### Requirement: Integration with Existing Pipeline

Cross-language dependencies MUST merge into the main dependency list before rule evaluation. All existing rules (`Cannot`, `MustDependOn`, `OnlyDependsOn`) and layer-matching logic SHALL apply without modification.

#### Scenario: Cross-language dep triggers rule

- GIVEN a rule `{Cannot: {source: "domain", target: "proto-generated"}}`
- AND a cross-language dep `proto/order.proto â†’ internal/pb/order.pb.go` where `internal/pb/` matches the `proto-generated` layer
- WHEN rules are evaluated
- THEN this violation appears in the check output

#### Scenario: Layer matching on generated paths

- GIVEN layer patterns `proto: "proto/**"` and `proto-generated: "internal/pb/**"`
- AND the cross-language dep `proto/order.proto â†’ internal/pb/order.pb.go` is emitted
- WHEN layer resolution runs
- THEN `SourceFile` resolves to `proto` layer
- AND `ImportPath` resolves to `proto-generated` layer
