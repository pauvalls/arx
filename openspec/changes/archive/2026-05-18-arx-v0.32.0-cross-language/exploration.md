## Exploration: Cross-Language Dependency Resolution

### Current State

Arx detects dependencies **within each language independently** and merges them into a flat list. The architecture:

1. **`ports.Detector` interface** (`internal/ports/detector.go`): Three methods ‚Äî `Name()`, `Detect(ctx, projectRoot)` (checks applicability), `ExtractImports(ctx, projectRoot, layers)` (returns `[]domain.Dependency`)

2. **Registry** (`internal/infrastructure/detector/registry.go`): Returns 10 standalone detectors ‚Äî Go, TypeScript, Python, Java, Kotlin, Rust, C#, Ruby, Swift, PHP. All run concurrently via `errgroup` in `application.RunDetectors()`.

3. **Dependency model** (`internal/domain/dependency.go`):
   ```go
   type Dependency struct {
       SourceFile    string  // e.g., "internal/service/order.go"
       SourceLine    int
       ImportPath    string  // e.g., "github.com/example/proto/order.pb.go"
       ResolvedLayer string  // which layer the import targets
   }
   ```
   No language metadata on source or target. No "origin" field (where the dependency truly comes from).

4. **Layer model** (`internal/domain/layer.go`): Pure path-pattern matching (glob). Language-agnostic by design.

5. **Rule evaluation** (`internal/domain/audit.go`): Iterates all dependencies √ó all rules. Each rule checks `sourceLayer` (resolved from `SourceFile` path) ‚Üí `targetLayer` (= `ResolvedLayer`). No language awareness in rules.

6. **The multi-language test fixture** (`test/fixtures/multi-language/`) tests Go + Python + TypeScript **in a single project** using glob-path layers. But it only tests that the Py/TS detectors run alongside the Go detector ‚Äî there is NO cross-language dependency tracing.

**Key limitation**: If a Go file imports `github.com/example/proto/order.pb.go`, the Go detector sees this as a local module import and resolves it to whatever layer contains `proto/order.pb.go`. But there is NO knowledge that this file was **generated from** `proto/order.proto`. The proto contract definition is invisible to arx.

### Patterns Analyzed

#### 1. Proto ‚Üí generated code

- **Detectability**: Easy
- **Signal**: File naming conventions (`foo.proto` ‚Üí `foo.pb.go`, `foo_pb2.py`, `foo.pb.ts`). Generated file headers (`// Code generated by protoc-gen-go`, `// @generated`). Proto import statements (`import "google/protobuf/timestamp.proto"`). `protoc` invocations in build files (`Makefile`, `buf.gen.yaml`, `buf.yaml`).
- **False positives**: Files named coincidentally `*pb.go` without being generated (rare). Generated files that have been manually modified (loses traceability). Proto files that are only imported by other proto files (no generated code consumed).
- **Detection mechanism**: 
  - **Naming-based scan**: Walk project globbing `**/*.pb.go`, `**/*_pb2.py`, etc. Apply configured mapping (`*.pb.go` ‚Üí `*.proto`).
  - **Header-based scan**: Read first 5 lines of generated files for code-gen markers.
  - **Build config**: Parse `buf.gen.yaml`, `Makefile` `protoc` commands for explicit mappings.
- **Confidence**: High for naming+header combo. Medium for naming alone.

#### 2. OpenAPI ‚Üí HTTP clients

- **Detectability**: Medium
- **Signal**: OpenAPI specs (`openapi.yaml`, `openapi.json`, `swagger.yaml`). Generated clients (`api_client.go`, `api.ts`, `*Client.js`). Code-gen comments at file headers. Build tool references (oapi-codegen, openapi-generator, swagger-codegen).
- **False positives**: HTTP client files that are hand-written but match naming patterns. Multiple versions of generated clients from different spec versions.
- **Detection mechanism**: Similar to proto ‚Äî naming conventions + header scanning. But less consistent naming across projects (client file names vary widely).
- **Confidence**: Medium. Highly dependent on user-provided naming conventions.

#### 3. REST API contracts (service A calls service B)

- **Detectability**: Hard
- **Signal**: HTTP client code in one service referencing URLs/hosts that match routes defined in another service. URL path patterns, `@FeignClient` annotations, service discovery names.
- **False positives**: External third-party API calls that happen to share paths. Proxied/redirected endpoints. Different environments with different URL patterns.
- **Detection mechanism**: Would require either:
  - Runtime traces (out of scope for static analysis)
  - Explicit config where users declare "service A depends on service B's API"
  - URL path extraction from HTTP client code + matching against route registrations
- **Confidence**: Low for automatic detection. High only with explicit configuration.

#### 4. Configuration sharing

- **Detectability**: Medium
- **Signal**: Multiple services importing/reading the same config file. Shared environment variable names. Common config paths (`/etc/app/config.yaml`, `config/defaults.yaml`).
- **False positives**: Coincidental same-name files with different content. Libraries that happen to read common paths.
- **Detection mechanism**: File-path intersection analysis ‚Äî which files are read/imported by multiple services? Then content-hash comparison to verify they're the same file.
- **Confidence**: Medium. File-level deps are easy, but verifying "same config" needs content hashing.

#### 5. Database migrations ‚Üí ORM models

- **Detectability**: Medium-Hard
- **Signal**: Migration files (`*.sql`, `*_migration.sql`, `V*__*.sql`). ORM model struct definitions matching table naming conventions. ORM tags/annotations referencing table names. Migration-to-entity correlation via naming conventions (`users` table ‚Üí `User` struct, `users` ‚Üí `User.js`).
- **False positives**: Table names that happen to match model names by coincidence. Generic field names. M2M join tables that have no direct model.
- **Detection mechanism**: Parse migration SQL for `CREATE TABLE` statements. Extract table and column names. Scan ORM models for struct/class names and field annotations. Match table names to model names via naming convention conversion (snake_case ‚Üî CamelCase).
- **Confidence**: Low-Medium automatic. Would benefit greatly from user-provided mapping conventions.

#### 6. Shared message schemas (Avro/Protobuf for queues)

- **Detectability**: Medium (similar to proto detection)
- **Signal**: Schema files (`*.avro`, `*.proto` with `syntax = "proto3"` for message-only files). Generated message classes in consumer code. Schema registry references in config.
- **False positives**: Proto files that define services (not just messages) ‚Äî already covered by proto pattern. Schema files that are registered but never consumed.
- **Detection mechanism**: Same as proto detection ‚Äî naming conventions + generated file analysis. Needs schema-registry awareness for full detection in async scenarios.
- **Confidence**: Medium. Schema file detection is easy; consumption detection needs consumer code analysis.

### Affected Areas

- `internal/ports/detector.go` ‚Äî **May need extension**: Add optional `DetectCrossLanguage` method or a new separate interface for cross-language detection. Current interface is language-scoped.
- `internal/infrastructure/detector/registry.go` ‚Äî **New cross-language detector registration**: Either a new detector type or a separate collection of cross-language resolvers alongside the per-language ones.
- `internal/domain/dependency.go` ‚Äî **Dependency struct expansion**: Add `SourceLanguage`, `TargetLanguage` fields, and an `Origin` field (e.g., "proto://proto/order.proto") to trace generated code back to its definition.
- `internal/domain/layer.go` ‚Äî **Layer tags**: Tags already exist but unused by detectors. Could add a `languages` filter to restrict which language's files a layer matches.
- `internal/application/check.go` ‚Äî **Integration point**: After all per-language detectors run, invoke cross-language phase. This may need a new `RunDetectorsCrossLanguage()` call in `runCheckWithService()`.
- `internal/application/service.go` ‚Äî **CheckService extension**: New method `DetectCrossLanguage()` or integration into existing `DetectCached()`.
- `internal/application/audit.go` ‚Äî **AuditService extension**: Cross-language deps in coupling matrix and debt scoring.
- `cmd/arx/check.go` ‚Äî **CLI implications**: New config loading for cross-language rules.
- `arx.yaml` ‚Äî **New config section**: `cross_language` mapping rules:
  ```yaml
  cross_language:
    mappings:
      - name: protobuf-go
        generated_by: "protoc-gen-go"
        source_pattern: "proto/**/*.proto"
        generated_pattern: "**/*.pb.go"
      - name: openapi-typescript
        source_pattern: "api/**/openapi.yaml"
        generated_pattern: "**/*api_client.ts"
  ```
- `internal/domain/config.go` ‚Äî **Config model extension**: New `CrossLanguageMappings` field in `Config` struct.
- `test/fixtures/multi-language/` ‚Äî **Test fixture extension**: Add proto files and generated pb.go files to test cross-language resolution.
- `internal/infrastructure/detector/shared/` ‚Äî **Shared pattern matching**: Glob-to-path matching for cross-language file pairs (similar to existing `glob.go` for import-to-layer matching).

### Recommended MVP Scope

**Build: Proto ‚Üí generated code detection only.**

Rationale:
1. **Strongest signal**: Clear naming conventions + header markers make false positives rare.
2. **Highest value**: Proto-driven polyglot architectures are the most common cross-language pattern in production (gRPC, buf, connect-go).
3. **Cleanest implementation path**: File-naming + header-scanning requires no new complex parsing.
4. **Extensible foundation**: The cross-language mapping config format can later accept OpenAPI, Avro, etc. without architectural changes.

**Defer:**
- OpenAPI ‚Üí HTTP clients (until we have user demand and more naming convention data)
- REST API contract detection (needs runtime traces or explicit route declarations)
- DB migrations ‚Üí ORM models (high complexity, low signal-to-noise)
- Configuration sharing (edge case, hard to automatically verify)

### Approach Options

#### 1. Post-processing phase (recommended)

Add a new detection phase that runs AFTER all per-language detectors complete. This phase:
- Scans for config-defined mappings (`source_pattern` ‚Üí `generated_pattern`)
- For each match pair, generates synthetic `Dependency` records from source ‚Üí generated file
- These deps join the existing pool and flow through the same rule-evaluation pipeline

**Pros:**
- Zero changes to existing detectors (no regression risk)
- Fits the current architecture: `RunDetectors` ‚Üí `RunCrossLanguageDetectors` ‚Üí `EvaluateRules`
- Users opt-in via config; no performance impact for single-language projects
- Mapping patterns are reusable across many projects

**Cons:**
- The current `Dependency` struct has no "synthetic" flag ‚Äî rules can't distinguish cross-language deps from regular imports without new metadata
- Doesn't handle the "reverse" case well: a Go file imports `order.pb.go`, the **true** dependency is `order.proto`. The synthetic dep would be `order.proto` ‚Üí `order.pb.go`, but the Go detector already created `service.go` ‚Üí `order.pb.go`. The cross-language context is in the relationship between the Go dep and the proto dep.

**Effort:** Medium (3-5 days)

#### 2. New detector type: `CrossLanguageDetector`

Add a second interface alongside the existing per-language one:

```go
type CrossLanguageDetector interface {
    Name() string
    DetectMappings(ctx context.Context, projectRoot string, config domain.CrossLanguageConfig) ([]CrossLanguageMapping, error)
}
```

A new registry entry runs these after standard detectors. Each mapping creates synthetic deps.

**Pros:**
- Clean separation of concerns (standard vs cross-language detection)
- Interface can grow independently (e.g., add `CheckConsistency()` later)
- Registry pattern matches existing architecture

**Cons:**
- More interfaces = more wiring in `CheckService` and `AuditService`
- The dependency between a Go import and its proto source is a **relationship** between two existing Dependency objects ‚Äî not a new standalone dependency
- Need to answer: how do we represent this in violations?

**Effort:** Medium (4-6 days)

#### 3. Extend `Dependency` model with metadata + enrich in post-processing

Instead of synthetic deps, add a `CrossLanguageOrigin` field to `Dependency`:

```go
type Dependency struct {
    SourceFile          string
    SourceLine          int
    ImportPath          string
    ResolvedLayer       string
    SourceLanguage      string             // NEW: "go", "python", etc.
    TargetLanguage      string             // NEW: language of resolved target
    CrossLanguageOrigin *CrossLangOrigin   // NEW: optional, if this dep crosses language boundaries
}

type CrossLangOrigin struct {
    SourceFile  string // e.g., "proto/order.proto"
    SourceType  string // "protobuf", "openapi", "avro"
    Description string // human-readable
}
```

Then in post-processing, for each existing dependency whose target is a generated file, look up the source definition and attach the origin.

**Pros:**
- Richer data model ‚Äî cross-language context is metadata on the existing dep, not a new dep
- No synthetic deps that might confuse coupling/debt calculations
- Violations can include cross-language context in their message

**Cons:**
- Changes to the core domain model affect serialization (JSON/YAML output, cache, history)
- Existing integrations (GitHub annotations, HTML reports, SARIF) need updates to display the new fields
- More invasive change than approach 1

**Effort:** Medium-High (5-8 days)

### Recommendation

**Approach 1 with a minimal Dependency extension.**

Specifically:
1. Add a `Language` field to `Dependency` for the source and target ‚Äî just a simple empty string that's filled by each detector (easy: detectors already know their own language).
2. Create `internal/infrastructure/detector/crosslanguage/` package with a `Detector` that:
   - Reads cross-language mappings from config
   - Scans for file pairs matching source/generated patterns
   - For each pair, emits a synthetic `Dependency` with `SourceFile = proto_path`, `ImportPath = generated_path`, `ResolvedLayer = <layer of generated>`
3. Wire this into `RunDetectors()` in `application/check.go` as a final step after per-language detectors
4. Update `Config` model to accept `cross_language` mappings

This approach:
- **Isolates risk** to a single new package
- **Reuses existing rule pipeline** ‚Äî users write rules like `Cannot: domain ‚Üí proto-generated` using layers
- **Needs zero changes** to the 10 existing detectors
- **Fits in a single PR** under 400 lines (new package + config + wiring)

The synthetic Dependency approach means a Go file in `domain/` that imports `order.pb.go` creates:
- (from Go detector): `domain/service.go ‚Üí pb/order.pb.go` (resolves to `proto` layer)
- (from cross-language detector): `proto/order.proto ‚Üí pb/order.pb.go` (also resolves `pb/order.pb.go` to `proto` layer)

The rule `domain cannot depend on proto` already catches the first one. The second one gives users visibility into the proto contract source.

### Risks

- **False positives from naming**: A file named `helper.pb.go` that isn't generated by protoc would create a non-existent link. Mitigation: header scanning verification (check first 5 lines for `Code generated by` comment).
- **Layer resolution ambiguity**: Generated files might not match any configured layer. Mitigation: clear error/ warning in detector output, or a default "generated" layer tag.
- **Performance**: The post-processing phase scans the entire project for generated files. Mitigation: only scan directories matching `generated_pattern` (globbing is cheap). Cache file-mapping results.
- **Config complexity**: Users need to understand file naming conventions to configure mappings. Mitigation: provide sensible defaults (pb.go ‚Üî proto, _pb2.py ‚Üî proto, swagger/ oapi patterns) that work out of the box.
- **Proto import chains**: Proto files import other proto files. A `foo.proto` that imports `bar.proto` creates an indirect dependency chain. Mitigation: MVP ignores proto-to-proto imports; document as future work.
- **Monorepo vs multi-repo**: In a monorepo, proto files and generated code live in the same project. In multi-repo, generated code is pulled in via package manager ‚Äî arx can't trace it without the source repo path. Mitigation: support `remote_source` config for mapping external module paths back to source definitions.

### Ready for Proposal

Yes. The exploration has identified:
- A clear MVP (proto ‚Üí generated code via naming conventions + header scanning)
- A recommended approach (post-processing phase with synthetic deps)
- Low risk to existing code
- Extensible foundation for future cross-language patterns

The proposal should focus on:
1. Cross-language mapping config schema for `arx.yaml`
2. The `CrossLanguageDetector` package implementation
3. Integration into the existing check/audit pipeline
4. Sensible defaults for common patterns (protobuf, OpenAPI, gRPC)
5. Documentation and test fixtures
