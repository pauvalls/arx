# Proposal: Cross-Language Dependency Resolution

## Intent

Arx detects dependencies per-language but cannot trace cross-language contracts (e.g., `proto/order.proto` → `internal/pb/order.pb.go`). This blind spot means rules like `domain cannot depend on proto` miss generated code in `internal/pb/` that's effectively a proto contract. Users in polyglot monorepos (Go + proto + TypeScript) need visibility into generated-code origins.

## Scope

### In Scope
- New `cross_language` config section in `arx.yaml` with mapping rules
- `CrossLanguageDetector` running as a post-processing phase after per-language detectors
- `Language` field on `Dependency` (filled by each detector)
- Synthetic `Dependency` records: proto source → generated file
- Proto header scanning (`// Code generated by`) to verify generation
- Config examples with sensible proto defaults
- Test fixtures: `test/fixtures/multi-language/` with proto + generated Go

### Out of Scope
- OpenAPI → HTTP client detection
- REST API runtime tracing
- Multi-repo generated code resolution
- Proto import chains (a.proto → b.proto)

### Delivery Strategy
`exception-ok` — single PR under 400 lines, low risk.

## Capabilities

### New Capabilities
- `cross-language-detection`: Config-driven mapping of IDL/generation sources to generated files. Post-processing phase emits synthetic deps linking proto sources to generated artifacts.

### Modified Capabilities
None — all capabilities are new.

## Approach

Post-processing phase (exploration approach 1):
1. Add `Language` string field to `domain.Dependency` — zero-cost, backward compatible
2. New `internal/infrastructure/detector/crosslanguage/` package:
   - `Detector` struct with `Detect(ctx, root, config) → []Dependency`
   - Reads `cross_language.mappings[]` from config
   - Globs `source_pattern` (e.g., `proto/**/*.proto`), matches against `generated_pattern` via naming conventions
   - Verifies generated files via header comment scan (first 5 lines)
   - Emits synthetic `Dependency{SourceFile: proto_path, ImportPath: pb.go_path, ResolvedLayer: <layer>}`
3. New `ports.CrossLanguageDetector` interface (or extend existing `CheckService.DetectCrossLanguage()`)
4. Wire into `runCheckWithService()`: `per-language deps → cross-language deps → evaluate`
5. New `domain.CrossLanguageConfig` / `CrossLanguageMapping` structs in `Config`

### Config Schema
```yaml
cross_language:
  mappings:
    - name: protobuf-go
      source_pattern: "proto/**/*.proto"
      generated_pattern: "**/*.pb.go"
```

Defaults: `proto/**/*.proto` ↔ `**/*.pb.go` (Go), `**/*_pb2.py` (Python), `**/*.pb.ts` (TypeScript).

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| False positive from naming (fake `*.pb.go`) | Low | Header scan verification as gate |
| Layer resolution ambiguity for generated files | Low | Clear warning in output; default "generated" tag |
| Performance: glob + header scan on large projects | Low | Only scan dirs matching `generated_pattern` |

## Rollback Plan

Revert the single PR. The change is isolated to the new package + config model + one integration point in `check.go`. No existing detector behavior changes.

## Dependencies

None — pure internal change.

## Success Criteria

- [ ] `arx check` detects `proto/order.proto → internal/pb/order.pb.go` as a cross-language dependency
- [ ] Generated file header scan correctly identifies protoc-generated files
- [ ] Existing single-language projects show zero change in output (backward compatible)
- [ ] All existing tests pass
