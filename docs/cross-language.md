# Cross-Language Dependency Resolution

Arx detects dependencies **across language boundaries**, not just within each language. This is what makes arx unique — no other tool traces how a `.proto` definition in one project area generates code consumed in another.

## How It Works

After all per-language detectors run, arx runs a **cross-language post-processing phase**. It reads config-defined mappings, matches source definition files to generated files, verifies generated-file headers, and emits synthetic dependencies.

```
Proto files ──→ CrossLanguageDetector ──→ Synthetic Dependency{Language:"cross"}
 OpenAPI     ──→                                      │
                                                      ▼
                                            Integrated into check pipeline
                                            Rules apply as usual
```

## Configuration

### Proto → Generated Code (stem matching)

```yaml
cross_language:
  mappings:
    - source_pattern: "proto/**/*.proto"
      generated_pattern: "internal/pb/**/*.pb.go"
      language: "go"
```

This matches `proto/order.proto` → `internal/pb/order.pb.go` by **stem** (same base filename). It also verifies the generated file has a `protoc-gen-` or `@generated` header.

### OpenAPI → HTTP Client (glob matching)

```yaml
cross_language:
  mappings:
    - source_pattern: "specs/openapi.yaml"
      generated_pattern: "internal/client/**/*.go"
      language: "go"
      match_strategy: "glob"
```

The `glob` strategy creates dependencies from **every source file to every generated file** (cartesian product). Use this when OpenAPI specs generate entire client directories without stem naming.

### Custom Header Patterns

```yaml
cross_language:
  mappings:
    - source_pattern: "schemas/**/*.avro"
      generated_pattern: "internal/models/**/*.go"
      language: "go"
      header_patterns:
        - "// Code generated by avro-gen"
        - "// auto-generated by schema-registry"
```

Custom `header_patterns` **override** the defaults rather than adding to them. Default patterns include:
- `// Code generated by protoc-gen-`
- `// @generated`
- `// Generated by OpenAPI Generator`
- `// auto-generated`
- `// Code generated by protoc-gen-grpc-gateway`

## Match Strategies

| Strategy | Behavior | When to Use |
|----------|----------|-------------|
| `stem` (default) | Match source → generated by same base filename (no extension) | Proto → single generated file per source |
| `glob` | Every source file depends on every generated file | OpenAPI → multiple generated files per spec |

## Example Rules

Once cross-language dependencies are detected, standard arx rules apply:

```yaml
layers:
  - name: domain
    paths: ["internal/domain/**"]
  - name: proto
    paths: ["proto/**"]
  - name: generated
    paths: ["internal/pb/**"]

rules:
  - id: domain-no-generated
    from: domain
    to: [generated]
    type: Cannot
    severity: error
    message: "Domain must not depend on generated proto code"
```

## Supported Patterns

| Source | Generated | Strategy | Header Check |
|--------|-----------|----------|-------------|
| `.proto` files | `.pb.go`, `*_pb2.py`, etc. | stem | ✅ `protoc-gen-`, `@generated` |
| `openapi.yaml` / `swagger.json` | Generated HTTP clients | glob | ✅ `OpenAPI Generator`, `auto-generated` |
| Custom generators | Any pattern | stem or glob | ✅ Configurable via `header_patterns` |

## Limitations (v0.33.0)

- **Proto imports**: `a.proto → b.proto` chains are not yet resolved (deferred)
- **Multi-repo**: Generated code from external packages not supported
- **Runtime traces**: REST API call detection requires runtime data (not planned)
